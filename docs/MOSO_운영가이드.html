0.한 페이지 요약 (꼭 알아야 할 것)

⭐ 핵심 정책 8가지 (2026-05-15 확정) MOSO 의 모든 매칭/추천/노출 동작은 아래 8개 규칙으로만 움직입니다. 이게 다입니다. 외워두시면 회원 응대가 쉬워집니다.
정책 1

매칭 영구 잠금

한 번이라도 매칭요청이 오간 (여성, 남성) 쌍은 status(대기·확정·거절) 와 무관하게 영구 잠금. 재요청 불가, 갤러리에서 영구 사라짐.

정책 2

노출기간 = 응답일 자정 + 7일

매칭 확정/거절된 카드는 응답일(KST) 자정 기준 다음 동일 요일 0시에 자동 사라짐. 대기중은 요청일 +30일.

정책 3

수락은 번복 불가, 거절은 7일 내 수락만 가능

남성 매칭 확정(approved)은 어떤 상태로도 못 돌림. 거절(rejected)은 7일 노출 기간 내 수락으로 번복 가능.

정책 4

거절은 영구

한 번 매칭이 성사되지 않은(rejected) 상대는 여성이 영구 재요청 불가, 관리자도 영구 MD 추천 불가.

정책 5

MD 추천 영구 잠금

한 번이라도 매칭이력 또는 MD 추천이 있는 (남, 여) 쌍은 관리자 모달 후보에서 자동 제외 + 서버 차단.

정책 6

인앱 브라우저 자동 안내

카카오톡/페북/인스타 안에서 들어오면 하단에 "다른 브라우저로 열어주세요" 안내. Android 는 Chrome 자동 실행.

정책 7

회원 만료일 자동 차단 (cron)

관리자 승인 시 자동 +30일 만료일 부여. 매일 cron 으로 만료 회원을 active → blocked 로 자동 전환.

정책 8

모든 API 권한 가드

모든 API 엔드포인트에 인증/관리자 가드. 외부 호출/봇 요청은 401/403 으로 즉시 차단 (2026-05-15 보안 핫픽스).

💡 운영팀 매뉴얼화 권장 위 8개 정책만 인쇄해서 책상에 두시면 회원 문의 응대 90% 가 해결됩니다. 뒤로 갈수록 깊이가 깊어지지만, 처음에는 이 페이지만 익히셔도 충분합니다.

1.회원 라이프사이클 & 상태 전이

회원은 4가지 상태(pending → active → blocked / rejected) 중 하나에 있습니다. 상태에 따라 사이트에서 할 수 있는 일이 정확히 결정됩니다.

stateDiagram-v2 [*] --> pending: 회원가입 신청 pending --> active: 관리자 승인
(만료일 자동 +30일) pending --> rejected: 관리자 반려
(사유 5자 이상 필수) rejected --> pending: 관리자 반려 취소
(사유 자동 비움) active --> blocked: 수동 차단
또는 만료일 도달 (cron) blocked --> active: 관리자 해제
상태이용 가능 화면매니저 액션
pending가입 신청 완료 안내만승인 또는 반려 (사유)
active모든 회원 기능 (성별별 메인, 매칭, 카트)차단
blocked이용 제한 안내만차단 해제
rejected반려 사유 안내만반려 취소 → pending 복귀
⚠️ 만료일 자동 차단 관리자가 회원을 승인하면 자동으로 30일 후 만료일이 부여됩니다. cron 이 매일 돌면서 만료일이 지난 active 회원을 자동으로 blocked 로 전환합니다. 만료일 연장은 관리자 회원 상세 페이지에서 가능.

3.화면별 기능 가이드

3-1. 랜딩 /

  • Google 로그인 버튼 (PKCE 직접 구현)
  • 이용약관 / 개인정보 링크
  • 인앱 브라우저(카카오톡/페북/인스타/라인/네이버)에서 진입 시 하단에 자동 안내 바
    • Android: "열기" 클릭 시 Chrome 자동 실행 (intent URL)
    • iOS: Safari 안내 모달 (Apple 정책상 자동 전환 불가)
  • 피드백용 미리보기 모드는 완전 제거됨 (2026-05-15)

3-2. 회원가입 /register (2단계)

1단계 — 프로필 기본정보

  • 본명 / 닉네임 (실시간 중복 체크, 2~10자, 한글·영문·숫자만) / 성별 / 출생년도 / 키 / 전화번호
  • 거주지 (시·도 + 구·군·시 동적) / 직장(11종) → 직업(직장별 동적) / 근무패턴 / 연봉 / 학력
  • 흡연 / MBTI / 매력 (100~200자) / 연인이 생기면 (100~200자)

사진 업로드 (필수)

  • 대표사진 2~4장 / 매력 사진 1장(1:1 크롭) / 연인이 생기면 사진 1장(1:1 크롭)
  • 서버: sharp WebP 변환, 5MB 제한, 1200×1600 축소

2단계 — 이상형 정보

  • 선호 나이/키/거주지/직장/연봉/학력/흡연/MBTI (모두 다중 선택)
  • 최애 포인트 정확히 4개 (선택 순서 번호 부여)
  • 약관 동의: 이용약관·개인정보·만19세 (필수) + 마케팅(선택)

3-3. 여성 메인 /female

  • 남성 active 회원 카드 그리드 (1열/2열 토글, 무한스크롤)
  • 정보 필터(7개 다중) + 직장 1개 선택 시 직업 sub-filter
  • 하트 토글 → 카트에 담기/빼기
  • 매칭 이력 있는 남성은 영구 안 보임 (정책 1)
  • 헤더: 매칭 후보 카운트 + 햄버거 메뉴(사이드바)

3-4. 여성 → 남성 상세 /female/[id]

  • 대표사진 캐러셀 + 정보 + 매력/연인 사진 (라이트박스)
  • 하단 버튼은 잠금 상태에 따라 자동 분기:
    • 이력 없음: 매칭 후보에 담기 / 빼기 토글
    • 대기중: 이미 매칭요청을 보낸 상대입니다
    • 확정: 매칭이 확정된 상대입니다
    • 거절: 매칭이 성사되지 않은 상대입니다

3-5. 매칭 후보 (카트) /female/cart

  • 상단: 카트에 담긴 남성 리스트 (개별 X 제거 가능)
  • "N명에게 매칭 요청 보내기" 버튼 → 일괄 전송
  • 전송 시 영구 잠금된 상대는 자동 제외 + 안내 메시지
  • 전송 후 매칭요청 보낸 남성은 카트에서 자동 삭제
  • 하단 고정: "매칭 요청 보낸 남성" 가로 스크롤 (참고용)

3-6. 매칭 요청 결과 /female/requests

  • 탭: 전체 / 대기중 / 수락 / 거절 (각 카운트 표시)
  • 카드 상태별 안내:
    • 대기중: 노란 박스 "상대방의 응답을 기다리고 있어요"
    • 매칭 성사: 초록 박스 + 🎉 + "운영진이 카톡으로 연락처 전달 예정"
    • 거절: 회색 박스 + 위로 메시지
  • 응답 후 7일(자정 기준) 지나면 자동 사라짐 (정책 2)

3-7. 남성 메인 /male

  • 받은 매칭요청 + MD 추천 통합 카드 그리드
  • 상태 뱃지: 대기중(노랑) / 매칭요청 보냄(초록) / 거절됨(회색) / MD(보라)
  • 가시성: 대기중 30일, 응답 완료 7일 (자정 기준) 후 자동 사라짐 (정책 2)
  • EmptyState: 받은 매칭이 없을 때 유쾌한 안내 + 랜덤 팁

3-8. 남성 → 여성 상세 /male/[id]

  • 사진 캐러셀 + 정보 + 매력/연인 사진
  • 하단 버튼은 매칭 상태에 따라 자동 분기:
    • 대기중: 매칭 요청(주황, confirm 후 번복 불가) / 거절(회색)
    • 매칭요청 보냄: "매칭요청을 보냈습니다" 안내 박스
    • 거절됨: "수락으로 번복" 버튼 + 7일 안내
  • MD 추천 응답도 정상 동작 (이전엔 source 누락 버그로 항상 실패, 2026-05-15 수정)

3-9. 관리자 메인 /admin

  • 검색: 이름/전화번호 (2글자 이상)
  • 필터: 상태 / 성별 / 매칭 / MD 추천 이력 / 정보 / 이상형
  • 회원 행: 썸네일 + 이름·닉네임 + 뱃지(NEW/MD/상태/성별/매칭) + 요약 + 만료일
  • 액션: 상태별로 다른 버튼 (승인/반려/차단/해제/반려취소)
  • 승인 시 자동으로 +30일 만료일 부여
  • 반려 시 5자 이상 사유 필수 (회원 화면에 노출)

3-10. 관리자 회원 상세 /admin/[id]

  • 사진 관리: 대표 4장 + 매력 + 연인 (X 클릭 시 즉시 삭제 confirm)
  • 프로필 인라인 수정 (더블클릭 텍스트 / 드롭다운 셀렉트)
  • 이상형 정보 표시 (열람 전용)
  • 매칭 내역 (상대 사진/이름/상태/날짜)
  • MD 추천 (남성 회원만): 매칭 이력 있는 여성 자동 제외 + 서버 차단
  • 관리자 메모 CRUD (수정 시각 표시, 회원에게 비공개)
  • 만료일 캘린더 + 반려 사유 표시/수정

3-11. 외부 공개 프로필 /profile/[token]

  • 로그인 불필요, 토큰 기반
  • 5일 만료 + 최대 10회 접근
  • 전화번호/이메일 제외
  • 현재 토큰 생성 UI 는 별도 (필요 시 관리자가 API 직접 호출)

4.핵심 정책 상세 (8개)

정책 1. 매칭 영구 잠금 ⭐

한 번이라도 매칭요청이 발생한 (여성, 남성) 쌍은 status 와 무관하게 영구 잠금됩니다.

방어선위치동작
1차여성 갤러리 /female매칭 이력 있는 남성을 화면에서 영구 제외
2차여성 → 남성 상세 /female/[id]하단 버튼이 잠금 사유 안내로 변경
3차카트 추가 API POST /api/cart409 차단 + 한국어 에러
4차일괄 매칭요청 API POST /api/matchinsert 차단 (upsert 제거됨, blocked 응답)
⚠️ 마이그레이션 주의 이 정책 적용으로, 기존에 거절/확정 매칭이 있던 (여성, 남성) 쌍은 여성 갤러리에서 즉시 사라집니다. 정상 동작입니다.

정책 2. 노출 기간 = 응답일 자정 + 7일 ⭐

모든 노출 기간은 KST(한국 시간) 자정을 기준으로 계산됩니다.

상태노출 시작노출 종료예시
pending요청 시각요청일(KST) 자정 + 30일5/15 응답 → 6/15 0시 만료
approved응답 시각응답일(KST) 자정 + 7일수요일 매칭 → 다음 수요일 0시 만료
rejected응답 시각응답일(KST) 자정 + 7일수요일 거절 → 다음 수요일 0시 만료

정책 3. 수락은 번복 불가, 거절은 7일 내 수락만 가능

stateDiagram-v2 [*] --> pending: 여성이 매칭요청 pending --> approved: 남성 수락 (confirm 후, 영구 확정) pending --> rejected: 남성 거절 (rejection_logs +7d 쿨타임) rejected --> approved: 7일 내 수락으로 번복 (쿨타임 해제) approved --> [*]: 최종 (관리자 오프라인 매칭 진행)

정책 4. 거절은 영구

한 번 매칭이 성사되지 않은(rejected) (여성, 남성) 쌍은 영구 재요청 불가입니다. 여성이 다시 그 남성에게 매칭요청 보낼 수 없고, 관리자도 그 남성에게 그 여성을 MD 추천할 수 없습니다.

정책 5. MD 추천 영구 잠금

관리자 추천 모달에서 후보 여성 리스트 자동 제외 + 서버 차단 양쪽 처리. 다음 경우 잠김:

  • 그 (남, 여) 쌍에 매칭요청 row 가 있음 (status 무관)
  • 그 (남, 여) 쌍에 MD 추천 row 가 이미 있음

정책 6. 인앱 브라우저 자동 안내

카카오톡/페이스북/인스타그램/라인/네이버 인앱 브라우저로 진입한 사용자에게 자동 노출.

OS / 브라우저동작
Android · 카카오톡"열기" → 카카오 외부 열기 시도 → Chrome 자동 실행
Android · 기타 인앱"열기" → Chrome 자동 실행 (intent URL)
iOS · 모든 인앱안내 모달: "오른쪽 위 ··· → Safari로 열기" + URL 복사
일반 브라우저알림 바 표시 안 함

Google 로그인 버튼 클릭 시에도 동일한 가드가 작동합니다 (Google 403 페이지로 안 빠짐).

정책 7. 회원 만료일 자동 차단 (cron)

  • 관리자 승인 시 자동으로 +30일 만료일 부여
  • 매일 cron 이 만료일 도달한 active 회원을 blocked 로 자동 전환
  • cron 엔드포인트: GET /api/cron/auto-block
  • 주의: CRON_SECRET 환경변수가 미설정이면 cron 이 503 으로 비활성화됩니다.

5.매니저 FAQ & 응대 시나리오

Q1. 관리자 페이지 "매칭 있음" 5개 옵션의 정확한 의미가?

옵션판정 기준의미
매칭 있음total > 0매칭요청 이력 1건 이상 (대기/확정/거절 모두 포함)
매칭 확정 있음approved > 0남성이 수락한 매칭이 있는 회원
대기중 있음pending > 0아직 응답 안 한 매칭이 있는 회원
거절 있음rejected > 0남성이 거절한 매칭이 있는 회원
매칭 없음total === 0매칭요청 이력 0건

Q2. 회원이 "갤러리에서 그 남성이 갑자기 안 보여요" 라고 문의하면?

→ 그 (여성, 남성) 쌍에 매칭요청 이력이 있어서 영구 제외된 케이스입니다. 정책 1에 따라 정상 동작입니다. 한 번 거절 또는 확정/대기 중인 상대는 다시 갤러리에 안 나오게 설계했습니다.

Q3. "왜 매칭이 사라졌나요?" 문의가 오면?

→ 응답일(KST) 자정 + 7일 후 자동 사라짐 (정책 2). 예: 수요일에 매칭 → 다음 수요일 0시 만료. 이 정책 이전에는 upsert 버그로 진짜 데이터 손실이 있을 수 있었으나, 2026-05-15 이후로는 정책에 따른 정상 만료입니다.

Q4. 카톡으로 링크 받은 회원이 "Google 로그인이 안 돼요" 라고 하면?

→ Android: 하단 알림바에서 "다른 브라우저로 열기" 누르면 Chrome 으로 자동 전환됩니다. iOS: 안내 모달의 가이드대로 Safari 로 열어주시라고 안내. 이게 Apple/Google 정책상 카톡 안에서는 안 되는 게 맞습니다.

Q5. "매칭이 확정된 회원에게 다시 매칭요청을 보내고 싶어요"

→ 정책 1 (영구 잠금) 으로 불가능합니다. 한 번 확정된 매칭은 운영팀이 카톡으로 후속 안내하는 단계로 넘어갑니다.

Q6. "매칭 후보 카트에 같은 사람이 두 번 보여요"

→ 정책 1 적용 후로는 발생하지 않습니다. 갤러리에서 영구 제외 + 서버 차단으로 4중 방어. 만약 발생한다면 데이터 정합성 이슈이니 즉시 개발팀에 알려주세요.

Q7. 회원이 "프로필 수정하고 싶어요" 하면?

→ 사이드바 메뉴 "내 프로필 수정요청" 누르면 채널톡 (https://2o017.channel.io) 으로 연결됩니다. 운영팀이 받아서 관리자 화면에서 직접 수정.

Q8. 거절당한 회원이 "다시 한 번 기회를 주세요" 하면?

→ 정책 4 (거절 영구). 같은 (남, 여) 쌍은 재시도 불가입니다. 다른 회원과의 매칭은 그대로 가능합니다.
